5.1.1 Input Data Risks and Mitigations

The quality of training data is often the single largest determinant of ML model quality. Poor data leads to poor models regardless of the sophistication of the model architecture. Common input data risks include:

  • Sampling bias: the training data is not representative of the target population (e.g., data collected only from a specific geographic region is used to build a global model)
  • Labeling errors: incorrect or inconsistent labels degrade model learning
  • Data drift: the statistical properties of production data differ from those of training data, causing model degradation over time
  • Missing values: incomplete records can bias learned patterns or cause inference failures
  • Outliers: extreme values may distort model training or indicate data quality issues
  • Privacy violations: training data may contain personally identifiable information (PII) that should not be included
  • Class imbalance: significant differences in the number of samples per class can bias the model toward the majority class

Mitigation approaches:

  • Exploratory Data Analysis (EDA): systematic visual and statistical exploration of data distributions, missing value patterns, and outliers before training
  • Statistical tests: formal tests for distribution conformance, independence, and normality
  • Schema validation: automated checks on column types, value ranges, and uniqueness constraints
  • Automated data quality checks: pipelines with built-in monitoring for common data issues (tools: Great Expectations, dbt tests, TFX Data Validation)

5.1.2 Testing for Bias

Bias types in ML data:

  • Sampling bias: the data collection process systematically excludes or underrepresents certain subgroups
  • Labeling bias: human annotators apply labels inconsistently or in ways that reflect societal prejudices (e.g., criminal risk scoring biased against minorities)
  • Measurement bias: the features themselves contain systematic errors (e.g., a biometric sensor that performs less accurately for dark skin tones)
  • Aggregation bias: combining data from distinct subgroups in a single model when separate models would be more appropriate (Simpson's Paradox)

Fairness metrics:

  • Demographic parity (statistical parity): the proportion of positive predictions should be equal across groups. P(Ŷ=1 | group=A) = P(Ŷ=1 | group=B)
  • Equal opportunity: the true positive rate should be equal across groups. P(Ŷ=1 | Y=1, group=A) = P(Ŷ=1 | Y=1, group=B)
  • Predictive equality (equalized odds): both TPR and FPR should be equal across groups

Disparate impact analysis: measures whether decisions affect protected groups disproportionately. The "four-fifths rule" (80% rule) states that the selection rate for the disadvantaged group should be at least 80% of the selection rate of the advantaged group. Counterfactual testing swaps sensitive attributes and checks whether the model's decision changes.

Tools: Fairlearn (Microsoft), AI Fairness 360 (IBM), Google What-If Tool.

5.1.3 Data Pipeline Testing

An ML data pipeline transforms raw source data into model-ready features. Testing must validate every stage of this pipeline.

Pipeline stages and testing focus:

  • Ingestion: data is read from source systems — test for completeness, schema conformance, and connection reliability
  • Cleaning: missing values are handled, outliers are treated — test that cleaning rules are applied correctly and consistently
  • Transformation: normalization, encoding, aggregation — test that transformations produce the expected output for known inputs (unit tests)
  • Feature engineering: creation of new features from raw data — test that feature formulas are correct; test for data leakage (features containing target information)
  • Loading: features are stored in a feature store or training dataset — test for correctness of stored values, data volume, and freshness

Schema validation checks: column names and types match the expected schema; values are within valid ranges; uniqueness constraints are satisfied; referential integrity is maintained.

Data lineage and provenance tracking: recording the origin and transformation history of data so that defects can be traced to their source. Lineage tools: Apache Atlas, OpenLineage, DataHub.

MLデータパイプラインは生の元データをモデルが使える特徴量に変換します。このパイプラインの各ステージを検証するためのテストが必要です。

ステージテストの焦点テスト手法

5.1.4 Testing for Data Representativeness

A training dataset is representative when it adequately captures the diversity and statistical properties of the target population — i.e., the population the model will be applied to in production.

Methods for testing representativeness:

  • Stratified sampling verification: confirm that each relevant subgroup (age group, geographic region, device type, etc.) is proportionally represented in the training data
  • Kolmogorov-Smirnov (KS) test: a non-parametric statistical test comparing two empirical distribution functions to check whether training data and production data come from the same distribution
  • Subpopulation analysis: evaluate model performance separately for each identified subgroup to detect differential performance
  • Temporal representativeness: ensure the training data spans an adequate time period to capture seasonal, cyclical, and trend patterns present in production

When a significant distribution shift between training and production is detected, the model should be retrained on data that better represents the production environment.

5.1.4 データ代表性のテスト

訓練データセットが代表的（representative）であるとは、モデルが本番環境で適用される対象集団の多様性と統計的性質を十分に捉えていることを意味します。

5.1.5 Dataset Constraint Testing (K3)

Dataset constraint testing involves formally specifying constraints that a dataset must satisfy and then verifying that the dataset conforms to those constraints. The tester must identify which constraints apply to a given dataset specification and determine whether each constraint is violated.

Types of dataset constraints:

  • Completeness constraints: specified fields must have non-null values. Example: "The 'diagnosis' column must be populated for all records."
  • Consistency constraints: values must be within valid ranges or conform to specified formats. Example: "Patient age must be between 0 and 130."
  • Class balance constraints: the ratio between the most frequent and least frequent class must not exceed a specified threshold. Example: "No class may represent more than 70% of the dataset."
  • Temporal constraints: data must meet freshness requirements. Example: "All records must have a timestamp within the last 90 days."
  • Volume constraints: the dataset must contain at least a specified minimum number of samples. Example: "Each class must have at least 500 samples."

Worked example:

Dataset specification: a medical image dataset for a tumor classifier. Constraints defined: (1) minimum 1000 images per class; (2) no more than 60% from a single hospital; (3) all images must have a confirmed pathology label; (4) no images older than 5 years.

Dataset audit results: Class A has 1200 images (PASS), Class B has 380 images (FAIL — volume constraint violated). 72% of images are from Hospital X (FAIL — source diversity constraint violated). 23 images have "unconfirmed" labels (FAIL — completeness constraint violated). All images are within 5 years (PASS).

5.1.5 データセット制約テスト（K3 — 適用）

データセット制約テストとは、データセットが満たすべき制約を形式的に定義し、データセットがその制約に適合しているかを検証することです。与えられたシナリオに対してどの制約が適用されるかを特定し、各制約が違反されているかを判定できなければなりません（K3：適用）。

データセット制約の種類

制約種別定義例
完全性制約 指定フィールドにnull値があってはならない 「diagnosis列はすべてのレコードで値が入力されていること」
一貫性制約 値が有効範囲内または指定フォーマットに従っていること 「患者年齢は0〜130の範囲内であること」
クラスバランス制約 最多クラスと最少クラスの比率が指定閾値を超えないこと 「いずれのクラスもデータセット全体の70%を超えないこと」
時間制約（鮮度制約） データが鮮度要件を満たしていること 「すべてのレコードのタイムスタンプが過去90日以内であること」
ボリューム制約 データセットが指定された最小サンプル数を含んでいること 「各クラスは最低500サンプルを持つこと」

実践例：腫瘍分類器の医療画像データセット

データセット仕様（Specification）
  • クラスごとに最低1000枚の画像（ボリューム制約）
  • 単一病院からの画像が60%を超えてはならない（多様性制約）
  • すべての画像に確定済み病理ラベルがあること（完全性制約）
  • 5年以内の画像であること（時間制約）
制約監査結果判定
クラスAが最低1000枚1200枚合格
クラスBが最低1000枚380枚違反（ボリューム制約）
単一病院60%以下病院Xが72%違反（多様性制約）
確定済みラベルのみ23枚が「未確定」違反（完全性制約）
5年以内のデータ全画像が5年以内合格

5.1.6 Label Correctness Testing

Supervised ML models learn from labeled training data. If labels are incorrect or inconsistent, the model will learn incorrect patterns — the model can be no more accurate than the labels it was trained on. Label correctness testing aims to detect and quantify mislabeled data before or after model training.

Methods for label correctness testing:

  • Human review and audit: domain experts manually examine a random sample of labeled records to estimate the labeling error rate
  • Multiple annotation: the same item is labeled by multiple independent annotators. The label is accepted when annotators agree; disagreements indicate potential labeling difficulty or errors
  • Inter-annotator agreement (IAA): Cohen's kappa (κ) measures the agreement between two annotators beyond chance. κ < 0.60 generally indicates poor labeling consistency
  • Model confidence score analysis: after training, items where the model assigns low confidence to the labeled class are flagged for review — these may be mislabeled
  • Model loss analysis: items that generate high training loss are disproportionately represented among mislabeled data — training loss can be used to prioritize review of potential mislabeled points
  • Cross-validation inconsistencies: items that are consistently misclassified across all cross-validation folds are strong candidates for review

5.1.6 ラベル正確性テスト

教師あり学習モデルはラベル付きデータから学習します。ラベルが不正確または一貫性がなければ、モデルは誤ったパターンを学習します。モデルの精度はラベルの品質を超えることができません。ラベル正確性テストは、モデル学習の前後にラベルの誤りを検出・定量化することを目的とします。

ラベル正確性テストの手法

手法説明タイミング
人間によるレビュー・監査 ドメイン専門家がランダムサンプルを手動検査し、ラベルエラー率を推定 学習前・学習後
複数アノテーション（Multiple Annotation） 同一アイテムを複数の独立したアノテーターがラベル付け。一致はラベルを採用、不一致はレビュー対象 学習前
アノテーター間一致度（IAA）— Cohen's Kappa（κ） 偶然を超えた2人のアノテーター間の一致度を測定。κ < 0.60 は一般的に一貫性が低いことを示す 学習前
モデル確信度スコア分析 学習後、モデルがラベル付きクラスに低い確信度を付けるアイテムをフラグ立て — ラベル誤りの可能性 学習後
モデル損失分析 高い訓練損失を生成するアイテムはラベル誤りデータに偏って多い。損失でラベル誤り候補の優先度付けが可能 学習後
クロスバリデーション不一致 全クロスバリデーションフォールドで一貫して誤分類されるアイテムは強力なレビュー候補 学習後

Cohen's Kappa の解釈

κ の値一致度の解釈
< 0.20わずかな一致（Slight）
0.20 – 0.40軽度の一致（Fair）
0.40 – 0.60中程度の一致（Moderate）
0.60 – 0.80実質的な一致（Substantial）— 許容レベル
0.80 – 1.00ほぼ完全な一致（Almost Perfect）
試験のポイント ラベル正確性テストの手法は学習「前」と学習「後」に分類できます。学習前：人間レビュー・複数アノテーション・IAA。学習後：確信度スコア分析・損失分析・クロスバリデーション不一致。Cohen's κ < 0.60 が「ラベリング一貫性が低い」の目安として頻出です。
